```bash
bazel build //cmd/gossl
```

```bash
gossl cert list --db /path/to/prod.db
```

# CA Management

## Creating a Root CA

```bash
gossl ca create --cn "My Root CA"
```

The first CA created in a database is always the root (self-signed). Subsequent calls to `ca create` without `--ca` will automatically create intermediates signed by the root.

### Options

| Flag | Default | Description |
|---|---|---|
| `--cn` | (required) | Common name |
| `--db` | `gossl.db` | Database file path |
| `--days` | `3650` | Validity period (default 10 years) |
| `--key-algo` | `rsa` | Key algorithm: `rsa` or `ecdsa` |
| `--key-size` | `4096` | RSA key size in bits |
| `--curve` | `P-256` | ECDSA curve: `P-256`, `P-384`, `P-521` |
| `--ca` | (root CA) | Parent CA common name |

### Examples

```bash
# RSA root CA with 4096-bit key (default)
gossl ca create --cn "Production Root CA"

# ECDSA root CA
gossl ca create --cn "EC Root CA" --key-algo ecdsa --curve P-384
```

## Creating Intermediate CAs

Intermediate CAs are signed by a parent CA. Without `--ca`, the root CA is used as the parent.

```bash
# Intermediate signed by root (automatic)
gossl ca create --cn "Staging Intermediate"

# Intermediate signed by a specific CA
gossl ca create --cn "US-East Intermediate" --ca "Production Intermediate"
```

You can build arbitrary CA hierarchies:

```
Root CA
├── Staging Intermediate
├── Production Intermediate
│   ├── US-East Intermediate
│   └── EU-West Intermediate
```

## Listing CAs

```bash
gossl ca list
gossl ca list --db prod.db
```

Output shows each CA's type (Root/Intermediate), common name, serial, algorithm, and validity dates.

# Certificates

## Issuing Certificates

Issue a certificate signed by a CA in the database. The private key is generated and stored alongside the certificate.

```bash
gossl cert issue --cn "api.example.com" --ca "My Intermediate" --san "api.example.com,10.0.0.1"
```

Without `--ca`, the root CA signs the certificate directly.

### Options

| Flag | Default | Description |
|---|---|---|
| `--cn` | (required) | Common name |
| `--ca` | (root CA) | Signing CA common name |
| `--db` | `gossl.db` | Database file path |
| `--days` | `365` | Validity period |
| `--san` | | Subject alternative names (comma-separated DNS names and IPs) |
| `--key-algo` | `rsa` | Key algorithm: `rsa` or `ecdsa` |
| `--key-size` | `4096` | RSA key size in bits |
| `--curve` | `P-256` | ECDSA curve: `P-256`, `P-384`, `P-521` |

### Examples

```bash
# Server cert with multiple SANs
gossl cert issue --cn "api.example.com" --ca "Intermediate" \
  --san "api.example.com,api.internal.example.com,10.0.0.1"

# ECDSA client cert
gossl cert issue --cn "service-a" --ca "Intermediate" --key-algo ecdsa

# Short-lived cert (90 days)
gossl cert issue --cn "temp.example.com" --days 90
```

## Signing External CSRs

Sign a Certificate Signing Request generated outside of gossl. The CSR's public key is validated against minimum strength requirements (RSA >= 2048 bits, ECDSA P-256/P-384/P-521).

```bash
gossl cert sign request.csr --ca "My Intermediate" --out signed-cert.pem
```

### Options

| Flag | Default | Description |
|---|---|---|
| `--ca` | (root CA) | Signing CA common name |
| `--db` | `gossl.db` | Database file path |
| `--days` | `365` | Validity period |
| `--out` | (required) | Output PEM file path |
| `--force` | `false` | Overwrite existing output file |

The signed certificate is both written to the output file and stored in the database. Subject and SANs are taken from the CSR.

### Key Strength Requirements

| Algorithm | Minimum |
|---|---|
| RSA | 2048 bits |
| ECDSA | P-256, P-384, or P-521 |

CSRs with keys below these minimums are rejected.

## Listing Certificates

```bash
gossl cert list
gossl cert list --db prod.db
```

Shows common name, serial, algorithm, validity dates, and status (active/revoked).

## Revoking Certificates

```bash
gossl cert revoke --serial A1B2C3D4
```

Marks the certificate as revoked in the database. Use the serial number from `cert list`.

## Exporting Certificates

Export a certificate to a PEM file. Use `--chain` to include the full CA chain (intermediate + root).

```bash
# Export leaf cert only
gossl cert export --serial A1B2C3D4 --out cert.pem

# Export with full chain (leaf + intermediate + root)
gossl cert export --serial A1B2C3D4 --out bundle.pem --chain

# Overwrite existing file
gossl cert export --serial A1B2C3D4 --out cert.pem --force
```

## Inspecting Certificates

Display certificate details from a file, stdin, or the database.

```bash
# From file
gossl cert inspect cert.pem

# From stdin
cat cert.pem | gossl cert inspect -

# From database
gossl cert inspect --db gossl.db --serial A1B2C3D4
```

Output includes subject, issuer, serial, validity, SANs, public key algorithm, signature algorithm, and expiry status.

## Verifying Certificates

Verify a certificate against a CA bundle file.

```bash
gossl cert verify cert.pem --chain ca-bundle.pem
```

The chain file should contain all CA certificates (roots and intermediates) needed to build the trust chain. Self-signed root CAs in the chain file are treated as trusted roots; others are used as intermediates.

```bash
# Verify using exported chain
gossl cert export --serial A1B2C3D4 --out bundle.pem --chain
gossl cert verify cert.pem --chain bundle.pem
```

# HTTPS Echo Server

Start an HTTPS echo server for testing TLS and mTLS flows.

## Using Certificates from the Database

```bash
gossl serve --dn "api.example.com"
```

Looks up the certificate by common name in the database, loads the private key, and starts the server.

## Using Certificate Files

```bash
gossl serve --cert server.pem --pkey server-key.pem
```

`--db`/`--dn` and `--cert`/`--pkey` are mutually exclusive.

## Options

| Flag | Default | Description |
|---|---|---|
| `--db` | `gossl.db` | Database file path |
| `--dn` | | Certificate common name from database |
| `--cert` | | Certificate PEM file |
| `--pkey` | | Private key PEM file |
| `--port` | `8443` | Listen port |
| `--addr` | `0.0.0.0` | Listen address |
| `--mtls` | `false` | Require and verify client certificates |
| `-v` | `false` | Include request headers in response |

## Response Format

The server responds with JSON:

```json
{
  "uri": "/test?foo=bar",
  "method": "GET",
  "body": "",
  "peer": {
    "subject": "CN=client",
    "issuer": "CN=Intermediate",
    "serial": "A1B2C3D4",
    "expiry": "2025-03-26T00:00:00Z"
  }
}
```

- `uri`, `method`, `body` — always included
- `headers` — included with `-v`
- `form` — included for POST/PUT requests with form data
- `peer` — included when the client presents a TLS certificate

## Enabling mTLS

With `--mtls`, the server requires clients to present a valid certificate. When using `--db`, all CAs in the database are trusted for client verification.

```bash
# Start mTLS server using DB certs
gossl serve --db myca.db --dn "api.example.com" --mtls

# Test with a client cert
curl --cert client.pem --key client-key.pem --cacert ca-bundle.pem https://localhost:8443/
```

Without `--mtls`, the server accepts client certificates if offered but does not require them.

## Typical Workflow

```bash
# Set up CA and certs
gossl ca create --cn "Test Root CA"
gossl ca create --cn "Test Intermediate"
gossl cert issue --cn "localhost" --ca "Test Intermediate" --san "localhost,127.0.0.1"
gossl cert issue --cn "test-client" --ca "Test Intermediate"

# Start server
gossl serve --dn "localhost" --mtls -v

# In another terminal, export client cert and test
gossl cert export --serial <CLIENT_SERIAL> --out client.pem --chain
# (export key manually or use the cert from the DB)
curl --cert client.pem --key client-key.pem -k https://localhost:8443/test
```

# Troubleshooting mTLS Issues

Common scenarios engineers encounter when working with mTLS-enabled services, and how to diagnose them with gossl.

## Expired Certificate

**Symptom:** TLS handshake fails with "certificate has expired" or similar.

```bash
gossl cert inspect cert.pem
```

Check the `Not After` and `Status` fields. If expired, reissue:

```bash
gossl cert issue --cn "api.example.com" --ca "Intermediate" --san "api.example.com"
```

## Missing Intermediate in Chain

**Symptom:** Client gets "unable to verify the first certificate" even though the cert is valid.

```bash
# Check what the server is sending
gossl sclient api.example.com:443

# Look for gaps — if Chain Certificate [1] is missing, the server
# isn't sending the intermediate
```

Fix by exporting the full chain and configuring the server with it:

```bash
gossl cert export --serial <SERIAL> --out fullchain.pem --chain
```

## Wrong CA Trust Store

**Symptom:** "certificate signed by unknown authority."

```bash
# Verify against your CA bundle
gossl cert verify cert.pem --chain ca-bundle.pem

# Check what CAs the server presents
gossl sclient api.example.com:443 --show-certs
```

## CSR Has Wrong Key Size

**Symptom:** `cert sign` rejects the CSR.

```bash
gossl csr inspect request.csr
```

Check the `Public Key` field. RSA keys must be at least 2048 bits. ECDSA must use P-256, P-384, or P-521.


```bash
gossl sclient api.example.com:443 --cert client.pem --key client-key.pem
```

import (

//go:embed docs/*.md
var docsFS embed.FS

func newDocsCmd() *cobra.Command {
	var topic string

	cmd := &cobra.Command{
		Use:   "docs [topic]",
		Short: "Print built-in documentation (useful for LLM context)",
		Long: `Dump embedded user documentation to stdout. Without arguments, prints all docs.
With a topic, prints only that section.

Topics: getting-started, ca-management, certificates, csr, sclient, serve, troubleshooting

Designed to be piped into an LLM prompt for context:
  gossl docs | llm "how do I set up mTLS with gossl?"`,
		RunE: func(cmd *cobra.Command, args []string) error {
			if len(args) > 0 {
				topic = args[0]
			}
			return runDocs(topic)
		},
	}

	return cmd
}

func runDocs(topic string) error {
	entries, err := fs.ReadDir(docsFS, "docs")
	if err != nil {
		return fmt.Errorf("reading embedded docs: %w", err)
	}

	// Sort for consistent output
	sort.Slice(entries, func(i, j int) bool {
		return entries[i].Name() < entries[j].Name()
	})

	if topic != "" {
		// Find matching topic
		target := topic + ".md"
		for _, e := range entries {
			if e.Name() == target {
				return printDoc(e.Name())
			}
		}
		// Try partial match
		for _, e := range entries {
			if strings.Contains(e.Name(), topic) {
				return printDoc(e.Name())
			}
		}
		return fmt.Errorf("unknown topic %q. Available: %s", topic, listTopics(entries))
	}

	// Print all docs
	fmt.Println("# gossl Documentation")
	fmt.Println()
	fmt.Println("The following is the complete documentation for gossl, a simplified")
	fmt.Println("alternative to openssl for mTLS certificate management and troubleshooting.")
	fmt.Println()

	for _, e := range entries {
		if err := printDoc(e.Name()); err != nil {
			return err
		}
		fmt.Println()
		fmt.Println("---")
		fmt.Println()
	}

	return nil
}

func printDoc(name string) error {
	data, err := docsFS.ReadFile("docs/" + name)
	if err != nil {
		return fmt.Errorf("reading %s: %w", name, err)
	}
	fmt.Print(string(data))
	return nil
}

func listTopics(entries []fs.DirEntry) string {
	var topics []string
	for _, e := range entries {
		name := strings.TrimSuffix(e.Name(), ".md")
		topics = append(topics, name)
	}
	return strings.Join(topics, ", ")
}
