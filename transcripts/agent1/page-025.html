Assistant2026-03-26T22:23:31.647Z

A few things to check:

  1. Branch mismatch — Your default branch on GitLab might be main but you're pushing to master. The CI file needs to be on the branch GitLab is configured to use.

  2. Tag push — The build/release stages only trigger on semver tags. Did you push the tag?

git tag v0.1.0
git push origin v0.1.0

Creating a release through the GitLab UI doesn't always trigger the tag pipeline if the tag wasn't pushed via git first.

  1. CI not enabled — Check Settings > CI/CD > General pipelines in your GitLab project. Make sure CI is enabled and shared runners are available.

  2. Check pipeline page — Go to CI/CD > Pipelines in your project. If nothing shows, the .gitlab-ci.yml might not be on the default branch yet.

Which of these applies? The most common issue is #2 — the release was created via the UI but no tag was pushed from git.

    steps:
      - uses: actions/checkout@v4

      - uses: bazel-contrib/setup-bazel@0.14.0
        with:
          bazelisk-cache: true

          repository-cache: true
      - name: Validate semver tag

      - name: Build binaries
        run: |
          bazel build --stamp \
            //cmd/gossl:gossl_linux_amd64 \
            //cmd/gossl:gossl_linux_arm64 \
            //cmd/gossl:gossl_darwin_amd64 \
            //cmd/gossl:gossl_darwin_arm64
      - name: Collect artifacts
        run: |
          mkdir -p dist
          for target in linux_amd64 linux_arm64 darwin_amd64 darwin_arm64; do

              --output=starlark \
              --starlark:expr="target.files.to_list()[0].path" 2>/dev/null | grep "bazel-out")

          done
          cd dist && sha256sum gossl_* > checksums.txt
      - name: Create release
        uses: softprops/action-gh-release@v2
