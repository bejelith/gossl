1	# gossl
2	
3	A simplified alternative to openssl for engineers working with mTLS-enabled services.
4	
5	mTLS is used to authorize requests by trusting the identity associated with the peer certificate. Engineers need effective tools to troubleshoot customer authentication issues — incorrect CSRs, expired certificates, missing intermediates in TLS handshakes that prevent chain-of-trust verification. They also need to manage test CAs for backend system testing. Today, openssl handles this but is overcomplicated and hard to use for newcomers.
6	
7	gossl replaces the most common openssl workflows with a single binary backed by a single SQLite file. It offers a command-line interface for scripting and a terminal UI for interactive CA management.
8	
9	## Problems we're trying to solve
10	
11	We struggle with engineers not being able to troubleshoot mTLS issues and general TLS thoroubleshooting, few really understand and/or read the TLS protocol so we had to invest into a lot of OpenSSL wrapper tools to help them. OpenSSL does a lot of things really well but it's also really complex and hard to use for newcomers and the documentation is often lacking or confusing.
12	
13	The other problem what im trying to solve is educational, this tool should help practive and understand basic concepts without the need of knowing the tool specs by leaveraging LLMs.
14	
15	## Features
16

18

204	## Future Work
205	
206	- **Full X.509 Subject field support** — Currently only Common Name (`--cn`) is supported. Add flags for Organization (`--o`), Organizational Unit (`--ou`), Country (`--c`), State (`--st`), Locality (`--l`), and other standard Subject fields for CA and certificate creation.
207	- **Custom OID extensions** — Support adding arbitrary OID codes and values to certificates and CSRs (e.g. `--ext 1.2.3.4.5=value`), enabling custom certificate policies, proprietary extensions, and compliance with organization-specific PKI requirements.
208	- **Configurable key usage and extended key usage** — Allow specifying `--key-usage` (e.g. `digitalSignature,keyEncipherment`) and `--ext-key-usage` (e.g. `serverAuth,clientAuth,codeSigning`) flags on certificate creation and CSR signing, instead of using hardcoded defaults.
209	- **Charm terminal UI** — Interactive terminal UI (`gossl ui --db myca.db`) built with [Bubble Tea](https://github.com/charmbracelet/bubbletea) for browsing and managing the CA hierarchy, issuing certificates, and inspecting the trust chain visually.
210	- **CI pipeline for pull requests** — GitHub Actions workflow that runs `bazel test //...` on every PR and triggers AI-powered code review using Claude Code agents with the code-review skills.
211	- **Remote Bazel cache** — Set up remote caching (e.g. [BuildBuddy](https://www.buildbuddy.io/) free tier or [EngFlow](https://www.engflow.com/)) to speed up CI builds and share cache between developers.
212	- **CI test log archiving** — Upload Bazel test logs (`bazel-testlogs/`) as GitHub Actions artifacts on failure so they can be downloaded and inspected after a failed run.
213	- **ACME server** — Embed an ACME (RFC 8555) server that issues certificates from any intermediate CA in the database, enabling automated cert provisioning for test environments using standard ACME clients like certbot or Caddy.
214	- **Encryption-at-rest for private keys** — Private keys are currently stored as plaintext PEM in the SQLite database. A passphrase-based encryption layer or OS keychain integration would protect key material if the database file is compromised.

1	# gossl CLI Design Spec
2	
3	## Overview
4	
5	gossl is a simplified alternative to openssl for engineers working with mTLS-enabled services. It provides two core capabilities:
6	
7	1. **Troubleshooting** — inspect certs, CSRs, verify chains, probe live servers
8	2. **Test CA management** — create and manage a CA hierarchy (root CA, intermediates, leaf certs) backed by a single SQLite file
9	
10	The tool offers a Cobra-based CLI for text commands and a Bubble Tea TUI for interactive CA management.
11	
12	## Architecture
13	
14	```
15	cmd/gossl/              # binary entry point (thin — no business logic)
16	internal/
17	  ca/                   # CA + intermediate creation, cert issuance/revocation
18	  cert/                 # x509 parsing, inspection, chain verification
19	  csr/                  # CSR parsing + validation
20	  tls/                  # remote server probing (TLS handshake, chain retrieval)
21	  store/
22	    certstore/          # SQLite — certs, CAs, intermediates, metadata
23	    keystore/           # KV API — private keys (SQLite-backed for now)
24	```
25	
26	### Layer Rules
27	
28	- **CLI/TUI are thin.** They parse input and call into `internal/` packages. No business logic in `cmd/`.
29	- **Packages are independent.** `cert` does not import `ca`. `ca` imports `cert` and `store`.
30	- **Store is behind interfaces.** `certstore` and `keystore` interfaces are defined in `internal/ca/` (consumer side), implemented in `internal/store/`.
31	- **Nothing is exported.** All packages live under `internal/`. This is a CLI tool, not a library.
32	
33	## Command Surface
34	
35	### CA Management (DB-backed)
36	
37	| Command | Key Flags | Description |
38	|---|---|---|
39	| `gossl ca create` | `--cn`, `--db`, `--days`, `--key-algo`, `--key-size`, `--curve` | Create root CA, initialize DB file |
40	| `gossl ca list` | `--db` | Show root CA info |
41	| `gossl intermediate create` | `--cn`, `--db`, `--days`, `--key-algo`, `--key-size`, `--curve` | Create intermediate signed by root CA |
42	| `gossl intermediate list` | `--db` | List all intermediates |
43	| `gossl cert issue` | `--cn`, `--db`, `--intermediate`, `--days`, `--san`, `--key-algo`, `--key-size`, `--curve` | Issue cert signed by an intermediate |
44	| `gossl cert list` | `--db` | List all issued certs |
45	| `gossl cert revoke` | `--serial`, `--db` | Mark cert as revoked |
46	| `gossl cert export` | `--serial`, `--db`, `--out`, `--chain`, `--force` | Export cert PEM to file (optionally full chain) |
47	
48	### Troubleshooting (file / stdin / remote)
49	
50	| Command | Args / Flags | Description |
51	|---|---|---|
52	| `gossl cert inspect` | `<file\|->`, `--db --serial` | Display cert details: subject, issuer, SANs, expiry, key algo, fingerprint |
53	| `gossl cert verify` | `<file\|->`, `--chain <ca-bundle-file>` | Verify cert against a CA bundle file, report errors |
54	| `gossl csr inspect` | `<file\|->` | Display CSR details |
55	| `gossl server check` | `<host:port>`, `--sni`, `--timeout`, `--ca-file`, `--db` | Connect, show cert chain, verify trust, flag issues |
56	
57	### Echo Server
58	
59	| Command | Key Flags | Description |
60	|---|---|---|
61	| `gossl serve` | `--db`, `--dn`, `--port`, `--mtls`, `-v` | Start HTTPS echo server using cert from DB (by common name) |
62	| `gossl serve` | `--cert`, `--pkey`, `--port`, `--mtls`, `-v` | Start HTTPS echo server using cert/key files |
63	
64	- `--db`/`--dn` and `--cert`/`--pkey` are mutually exclusive flag groups.
65	- `--dn` matches against cert common names in the DB.
66	- Default port: 8443.
67	- Without `--mtls`: accepts client certs if offered, does not require them.
68	- With `--mtls`: requires and verifies client cert against the CA chain in `--db` (or system trust store for file mode).
69	- Response body: URI, form data, body. With `-v`: also includes request headers.
70	- If a client presents a cert, the response always includes parsed peer cert info (subject, issuer, serial, expiry).
71	
72	### TUI
73	
74	| Command | Flags | Description |
75	|---|---|---|
76	| `gossl ui` | `--db` | Launch Bubble Tea TUI for interactive CA management |
77	
78	### Input Conventions
79	
80	- All DB commands default to `--db gossl.db` in the current directory if not specified.
81	- Stdin is indicated by `-` or absence of a file arg when piped.
82	- `--key-algo` supports `rsa` (default) and `ecdsa`.
83	- `--key-size` defaults: RSA 4096. Ignored for ECDSA.
84	- `--curve` defaults: P-256. Ignored for RSA.
85	- `--days` defaults: CA 3650 (10 years), intermediate 1825 (5 years), cert 365 (1 year).
86	- `--timeout` defaults: 10s for `server check`.
87	- `--force` on export overwrites existing files. Without it, export fails if file exists.
88	- Output format is PEM. RSA keys use PKCS#1 encoding. ECDSA keys use SEC 1 (EC PRIVATE KEY PEM block).
89	- Serial numbers are uppercase hex without colons (e.g., `A1B2C3D4`).
90	
91	## Storage
92	
93	### SQLite Schema (certstore)
94	
95	One root CA per database. Multiple intermediates per CA. Multiple certs per intermediate.
96	
97	Schema version is tracked via SQLite `user_version` pragma. On open, gossl checks the version and refuses to open a mismatched DB with a clear error message. No automatic migrations in v1.
98	
99	```sql
100	PRAGMA user_version = 1;
101	
102	CREATE TABLE ca (
103	    id          INTEGER PRIMARY KEY CHECK (id = 1),
104	    common_name TEXT NOT NULL,
105	    serial      TEXT NOT NULL UNIQUE,
106	    key_algo    TEXT NOT NULL,
107	    cert_pem    TEXT NOT NULL,
108	    not_before  DATETIME NOT NULL,
109	    not_after   DATETIME NOT NULL,
110	    created_at  DATETIME NOT NULL
111	);
112	
113	CREATE TABLE intermediate (
114	    id          INTEGER PRIMARY KEY,
115	    ca_id       INTEGER NOT NULL REFERENCES ca(id),
116	    common_name TEXT NOT NULL,
117	    serial      TEXT NOT NULL UNIQUE,
118	    key_algo    TEXT NOT NULL,
119	    cert_pem    TEXT NOT NULL,
120	    not_before  DATETIME NOT NULL,
121	    not_after   DATETIME NOT NULL,
122	    created_at  DATETIME NOT NULL
123	);
124	
125	CREATE TABLE cert (
126	    id              INTEGER PRIMARY KEY,
127	    intermediate_id INTEGER NOT NULL REFERENCES intermediate(id),
128	    common_name     TEXT NOT NULL,
129	    serial          TEXT NOT NULL UNIQUE,
130	    key_algo        TEXT NOT NULL,
131	    cert_pem        TEXT NOT NULL,
132	    not_before      DATETIME NOT NULL,
133	    not_after       DATETIME NOT NULL,
134	    revoked_at      DATETIME,
135	    created_at      DATETIME NOT NULL
136	);
137	```
138	
139	### KeyStore (KV API)
140	
141	Private keys are stored separately behind a KV interface to allow future migration to Vault, KMS, or HSM.
142	
143	```go
144	type KeyStore interface {
145	    Store(ctx context.Context, id string, keyPEM []byte) error
146	    Load(ctx context.Context, id string) ([]byte, error)
147	    Delete(ctx context.Context, id string) error
148	}
149	```
150	
151	Keys are referenced by the serial number of the associated cert/CA/intermediate. The initial SQLite implementation uses a `(id TEXT PRIMARY KEY, key_pem BLOB)` table in the same DB file.
152	
153	`Delete` exists on the interface for future key rotation/destruction workflows. No CLI command calls it in v1.
154	
155	### Code Generation
156	
157	SQL queries are written in `.sql` files and compiled to Go code using sqlc. sqlc runs as a Bazel build step via multitool — generated code is not committed to the repository.
158	
159	```
160	internal/store/
161	  schema.sql      # CREATE TABLE statements
162	  queries.sql     # named queries
163	  sqlc.yaml       # sqlc configuration
164	```

1	# Troubleshooting
2	
3	## mTLS: Client Certificate Rejected
4	
5	**Symptom:** TLS handshake fails with `certificate required`, `bad certificate`, or `unknown CA`.
6	
7	**Causes and fixes:**
8	
9	1. The client certificate was not signed by a CA that the server trusts. Confirm the server is using a trust store that includes the issuing CA. In gossl serve, the trusted pool comes from the database — ensure the client cert was issued by a CA in that database.
10	
11	2. The client is not presenting a certificate at all. Check that `--cert` and `--key` are passed to the connecting tool.
12	
13	3. The client certificate has been revoked. Check `gossl cert list` and look for `revoked` status on the relevant serial.
14	
15	## Expired Certificate
16	
17	**Symptom:** TLS error `certificate has expired` or `x509: certificate has expired or is not yet valid`.
18	
19	**Fix:** Issue a new certificate:
20	
21	```
22	gossl cert issue --cn "service.example.com" --san "DNS:service.example.com" --ca "Issuing CA"
23	```
24	
25	Check expiry of any certificate:
26	
27	```
28	gossl cert inspect service.pem
29	gossl cert inspect --db gossl.db --serial 0A1B2C3D
30	```
31	
32	Check expiry of CA certificates:
33	
34	```
35	gossl ca list
36	```
37	
38	## Missing Intermediate CA
39	
40	**Symptom:** Chain validation fails even though the end-entity certificate is valid.
41	
42	**Symptom detail:** `x509: certificate signed by unknown authority` when the root is trusted but an intermediate is not in the presented chain.
43	
44	**Fix:** Export the full chain and use it when serving or connecting:
45	
46	```
47	gossl cert export --serial <end-entity-serial> --chain --out fullchain.pem
48	```

54	```

57

59

61

63	gossl sclient internal.corp:8443 --ca-file root-ca.pem
64	```
65	
66	Export the root CA certificate from the database:
67	
68	```
69	gossl cert export --serial <root-ca-serial> --out root-ca.pem
70	```
71	
72	The root CA serial is visible in `gossl ca list`.
73	
74	## Weak Key in CSR Rejected
75	
76	**Symptom:** `gossl cert sign` fails with a message about key strength.
77	
78	**Cause:** `cert sign` enforces a minimum RSA key size of 2048 bits. CSRs with 1024-bit RSA keys are rejected.
79	
80	**Fix:** Regenerate the CSR with a stronger key:
81	
82	```
83	openssl genrsa -out key.pem 2048
84	openssl req -new -key key.pem -subj "/CN=example.com" -out request.csr
85	gossl cert sign request.csr --ca "Issuing CA"
86	```
87	
88	Inspect the key size in an existing CSR before signing:
89	
90	```
91	gossl csr inspect request.csr
92	```
93	
94	## SNI Mismatch
95	
96	**Symptom:** Server presents a certificate for the wrong hostname, or `gossl sclient` shows a certificate whose subject does not match the host you connected to.
97	
98	**Cause:** The server is returning a default certificate because the SNI hostname in the ClientHello did not match any configured virtual host, or the certificate simply has the wrong SANs.
99	
100	**Fix — if connecting by IP:** Override SNI to send the expected hostname:
101	
102	```
103	gossl sclient 10.0.0.5:443 --sni api.example.com
104	```
105	
106	**Fix — if the certificate is missing SANs:** Reissue with correct SANs:
107	
108	```
109	gossl cert issue --cn "api.example.com" \
110	  --san "DNS:api.example.com,DNS:api-internal.example.com" \
111	  --ca "Issuing CA"
112	```
113	
114	Note: Modern TLS stacks match on SANs, not CN. Always include `DNS:` entries in `--san`.
115	
116	## Connection Timeout / Hang
117	
118	**Symptom:** `gossl sclient` hangs indefinitely.
119	
120	**Fix:** Set an explicit timeout:
121	
122	```
123	gossl sclient host:443 --timeout 5s
124	```
125

127

133

135

137	gossl cert issue --cn "example.com" --db /path/to/pki.db
138	gossl cert list --db /path/to/pki.db
139	```

# Private Keys

## Inspecting a Private Key

From a file:

```
gossl key inspect key.pem
```

From stdin:

```
cat key.pem | gossl key inspect -
```

Output for an RSA key:

```
Algorithm:         RSA
Key Size:          2048 bits
Public Key SHA256: SHA256:ab:cd:ef:12:34:56:78:...
```

Output for an ECDSA key:

```
Algorithm:         ECDSA
Curve:             P-256
Public Key SHA256: SHA256:ab:cd:ef:12:34:56:78:...
```

### Options

| Flag | Default | Description |
|------|---------|-------------|
| `--raw` | false | Show full modulus (RSA) or public point (ECDSA) in hex |

With `--raw`, an additional line is printed:

- RSA: `Modulus: <colon-separated uppercase hex>`
- ECDSA: `Public Point: <colon-separated uppercase hex>`

## Exporting a Private Key

Export the private key for a certificate by common name:

```
gossl key export --cn "server.example.com"
```

Write to a file:

```
gossl key export --cn "server.example.com" --out key.pem
```

Overwrite an existing file:

```
gossl key export --cn "server.example.com" --out key.pem --force
```

Without `--out`, the PEM key is written to stdout.

### Options

| Flag | Default | Description |
|------|---------|-------------|
| `--cn` | required | Certificate common name |
| `--db` | `gossl.db` | Path to the SQLite database |
| `--out` | stdout | Output file path |
| `--force` | false | Overwrite existing output file |

## Verifying a Key Matches a Certificate

Both `gossl key inspect` and `gossl cert inspect` print a `Public Key SHA256` line. If the fingerprints match, the key and certificate are a pair.

```
gossl cert inspect cert.pem
gossl key inspect key.pem
```

Compare the `Public Key SHA256` lines — if they are identical, the key matches the certificate.
